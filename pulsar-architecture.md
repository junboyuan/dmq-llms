# Pulsar 架构设计

## 整体架构

Apache Pulsar 采用分层架构设计，将计算与存储分离，实现了云原生的高可用消息平台。

```
┌──────────────────────────────────────────────────────────────────┐
│                          客户端层                                  │
│        Producer  │  Consumer  │  Reader  │  Admin CLI            │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│                        服务发现层                                 │
│                     ZooKeeper 集群                                │
│         (集群元数据、Broker 注册、负载均衡协调)                     │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│                          服务层                                   │
│                      Broker 集群                                  │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐         │
│    │ Broker 1│   │ Broker 2│   │ Broker 3│   │ Broker N│         │
│    │(无状态)  │   │(无状态)  │   │(无状态)  │   │(无状态)  │         │
│    └─────────┘   └─────────┘   └─────────┘   └─────────┘         │
│                                                                   │
│    职责：消息传递、协议转换、负载均衡、Topic 管理                    │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│                         持久化层                                  │
│                    BookKeeper 集群                                │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐         │
│    │ Bookie 1│   │ Bookie 2│   │ Bookie 3│   │ Bookie N│         │
│    │(存储节点)│   │(存储节点)│   │(存储节点)│   │(存储节点)│         │
│    └─────────┘   └─────────┘   └─────────┘   └─────────┘         │
│                                                                   │
│    职责：消息持久化、Ledger 管理、多副本存储                        │
└──────────────────────────────────────────────────────────────────┘
```

---

## 1. 分层架构详解

### 1.1 架构优势

| 特性 | 说明 |
|------|------|
| 服务存储分离 | Broker 和 BookKeeper 独立扩展 |
| 无状态服务层 | Broker 故障快速恢复 |
| 持久化存储层 | 数据高可靠保证 |
| 灵活扩展 | 各层独立水平扩展 |

### 1.2 与传统架构对比

**Kafka 架构**：
```
┌─────────────────────────────────────────┐
│               Broker 节点               │
│  ┌─────────────────────────────────┐   │
│  │         计算层 + 存储层           │   │
│  │    (消息传递 + 本地磁盘存储)      │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**Pulsar 架构**：
```
┌─────────────┐     ┌─────────────┐
│ Broker 节点  │ ──→ │ Bookie 节点  │
│  (计算层)   │     │  (存储层)   │
└─────────────┘     └─────────────┘
    可独立扩展          可独立扩展
```

### 2.2 Broker 核心组件

#### 协议处理器

| 协议 | 端口 | 用途 |
|------|------|------|
| Pulsar Binary | 6650 | 高性能二进制协议 |
| HTTP/REST | 8080 | 管理 API 和 Web 端点 |
| Kafka Protocol | 9092 | Kafka 兼容层（可选） |
| WebSocket | 8080/ws | WebSocket 连接 |

#### Topic Lookup

```
客户端请求 Topic
       │
       ▼
┌──────────────────┐
│  Broker 查找     │
│  (查询 ZooKeeper) │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  返回 Broker 地址 │
│  (负载最轻的)     │
└──────────────────┘
         │
         ▼
    客户端连接
    目标 Broker
```

#### 消息分发器

负责将消息推送给消费者：

```
┌─────────────────────────────────────────┐
│              Dispatcher                  │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │Consumer1│  │Consumer2│  │Consumer3│  │
│  │(Push)   │  │(Push)   │  │(Push)   │  │
│  └────┬────┘  └────┬────┘  └────┬────┘  │
│       │            │            │       │
│       ▼            ▼            ▼       │
│   消息队列     消息队列     消息队列      │
└─────────────────────────────────────────┘
```

### 2.3 Broker 无状态特性

- **无本地存储**：所有数据存储在 BookKeeper
- **快速故障恢复**：Broker 重启或迁移秒级完成
- **负载均衡**：Topic 自动在 Broker 间迁移

```
Broker 1 故障
       │
       ▼
┌──────────────────────┐
│   检测 Broker 故障    │
│   (ZooKeeper 心跳)   │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│   Topic 重新分配     │
│   (转移到其他 Broker)│
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│   客户端重连         │
│   (Lookup 新 Broker) │
└──────────────────────┘

恢复时间：< 30 秒
```

### 3.3 Ledger 生命周期

```
┌────────────────────────────────────────────────────────────┐
│                   Ledger 生命周期                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│   1. 创建 Ledger                                           │
│      │                                                     │
│      ▼                                                     │
│   ┌──────────────────────────────────────────────────┐    │
│   │  ZooKeeper 创建 Ledger 元数据                      │    │
│   │  • Ledger ID 分配                                  │    │
│   │  • 选择 Bookie 节点 (Ensemble)                     │    │
│   │  • 记录写入 Quorum 配置                            │    │
│   └──────────────────────────────────────────────────┘    │
│      │                                                     │
│      ▼                                                     │
│   2. 写入 Ledger (活跃状态)                                 │
│      │                                                     │
│      │   ┌─────────┐                                       │
│      │   │ Entry 1 │ ──→ Bookie 1, 2, 3                   │
│      │   │ Entry 2 │ ──→ Bookie 1, 2, 3                   │
│      │   │ Entry 3 │ ──→ Bookie 1, 2, 3                   │
│      │   │  ...    │                                       │
│      │   └─────────┘                                       │
│      │                                                     │
│      ▼                                                     │
│   3. 关闭 Ledger                                           │
│      │                                                     │
│      │   触发条件：                                        │
│      │   • Ledger 大小达到阈值                            │
│      │   • Ledger 时长达到阈值                            │
│      │   • Topic 卸载                                      │
│      │                                                     │
│      ▼                                                     │
│   4. Ledger 只读                                            │
│      │                                                     │
│      │   • 不可修改                                        │

│      │                                                     │

│      ▼                                                     │

│   │  清理 Ledger 数据                                  │    │

│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### Topic 归属查找

```python
# 客户端查找 Topic 所属 Broker

def lookup_topic(topic):
    # 1. 从 ZooKeeper 获取 Topic 所属 bundle
    bundle = get_bundle_for_topic(topic)

    # 2. 从 ZooKeeper 获取 bundle 所属 Broker
    broker = get_broker_for_bundle(bundle)

    # 3. 返回 Broker 地址
    return broker.service_url
```

## 8. 性能优化

### 8.1 Broker 调优

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| brokerServiceThreads | 处理消息的线程数 | CPU 核心数 × 2 |
| numHttpThreads | HTTP 服务线程数 | CPU 核心数 |
| maxConcurrentLookupRequest | 最大并发查找请求数 | 50000 |
| defaultNumberOfNamespaceBundles | 默认 Bundle 数量 | 4 |

### 8.2 BookKeeper 调优

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| journalSyncData | Journal 同步写入 | true |
| journalMaxGroupWaitMSec | Journal 批量写入延迟 | 1-2ms |
| numAddWorkerThreads | 添加工作线程数 | CPU 核心数 |
| numReadWorkerThreads | 读取工作线程数 | CPU 核心数 × 2 |

### 8.3 Pulsar Client 调优

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| ioThreads | IO 线程数 | CPU 核心数 |
| connectionsPerBroker | 每 Broker 连接数 | 1-4 |
| batchingMaxMessages | 批量发送消息数 | 1000 |
| batchingMaxPublishDelayMs | 批量发送延迟 | 1-10ms |

## 总结

Pulsar 架构核心特点：

1. **分层架构**：服务层（Broker）与存储层（BookKeeper）分离
2. **无状态 Broker**：快速故障恢复，水平扩展
3. **持久化存储**：BookKeeper 多副本保证数据安全
4. **高可用设计**：组件故障自动恢复
5. **云原生**：容器化部署友好，支持 Kubernetes
